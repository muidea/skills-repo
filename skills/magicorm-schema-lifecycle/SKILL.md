# MagicOrm Schema Lifecycle

Use this skill when the task is about table creation, table deletion, relation table layout, or why database schema does not match the model declaration.

## Scope

- `Create(entity)` and `Drop(entity)`
- Host table and relation table generation
- Type mapping and column selection
- Schema evolution review

## Core rules

- Schema structure must be derived from the active model declaration.
- Relation fields should generate relation storage, not ad hoc business tables.
- Type mapping must remain consistent with the declared field type.
- Schema fixes belong in `magicOrm`, not in project-specific SQL patches unless the project explicitly owns the database outside ORM control.

## Review workflow

1. Confirm the active model definition.
2. Check the generated host table shape.
3. Check relation tables for include or reference fields.
4. Verify type mapping for scalar, array, and relation fields.
5. Fix ORM schema builders before adjusting service code.

## Common bad patterns

- Hand-maintaining schema that should be generated by the model
- Letting relation storage drift from relation declaration
- Fixing type mismatches only in one database backend
- Treating schema symptoms as query bugs
